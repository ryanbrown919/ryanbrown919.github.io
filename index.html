ProSpike - Volleyball Setting Machine | Technical Project

Project Background

As part of Integrated Engineering, every year we form groups around an idea and complete a term-long project. We have a lot of flexibility with what we attempt to do, as long as we can show a solid game plan and market value for our project. For IGEN 330, our budget was $400

In my 3rd year, four of my peers and I decided to make a machine that can launch a volleyball in a similar fashion to a human setter. This would allow users to practice offensive drills, arguably the most fun aspect of the sport, without requiring a setter to be present. The project was named Prospike, and with 6 months until presenting at Design and Innovation day 2023, we commenced.

Project Description

There were many aspects of this project to consider, namely the aiming mechanism, launching mechanism, loading mechanism and the user interface.

Aiming

We prioritized the front row positions, so our design had a full range of motion along one axis of tilt, or else it can hit anywhere along the next, around two feet off of the netting. This was accomplished by a rotating frame built using alumnium extrusions and a linear actuator attached to one end of the frame. The linear actuator was controlled by an arduino running a P-Control script, given input from an ultrasonic sensor mounted to the linear actuator.

Launching

Launching was accomplished through two 3D printed wheels independantly connected to two 24V electric motors. These motors are contorlled by a off-the-shelf PWM controller. We modified this controller by replacing the dial potentiometer with an I2C digital potentiometer, allowing us to programmically control the speed of the motors based on the resistance specified.

Loading

To launch the ball, it was raised and pushed between the revolving wheels by a halo mounted to a waterjet cut rack, moved by a stepper rotating a spur gear. This enabled us to carefully lift the ball the exact right distance to force it between the wheels to launch. To reload a new ball, the basket was returned to the bottom and a servo blocking balls from a small ramp moves out of the way temporarily to allow a single ball to roll into the basket, ready to fire again.

User Interface

The user interface was powered by a HP Spectre computer set up in tablet orientation on a shelf on the machine. The interface was coded using C# and XAML, utilizing Windows Presentation Format and Visual Studio Blend to create an intuitive user interface. On the right side, several of the most common sets are listed as well as options for three custom sets, configurable in the settings page. On the left side, the display features a custom made volleyball net graphic with a live visual of the ball and its trajectory, including for the custom sets. Custom sets allow the user to specify the apex height and final potiion of the ball, controlled by a vertical and horizontal slider.

The user can select up to 8 sets then activate the play button, after which the machine will run through the specified sequence of sets. As the sequence plays, the next set is displayed on screen.

Personal Contributions

This project was more complex than anything I have done before and was a massive group effort. That being said, I did contribute to many important aspects of the project.

Aiming

Setting up the linear actuator and establishing the control was my contribution to the aiming mechanism. Due to our limited budget, the linear actuator we acquired did not contain positional feedback. My solution to this was to use an ultrasonic sensor, plentifully avalible for free in our workshop, mounted on the actuator body and a reference target mounted on the actuator arm to calculate the extension. I also developed the arduino code which received a distance in centimeters from another arduino through I2C and continously utilized P-control to reach the correct distance and keep the frame at the correct angle, or else accelerate the frame in the required direction. PID-control was investigated, but the speed of the linear actuator limited the use of such a responsive system so it was not required.

Launching

As I was heavily involved with system control, I implemented the replacement of manual speed control with a digital potentiomter. Also connected through I2C, 127 different resistance levels corresponded to different percentage powers and thus coould be used to control the trajectory of the ball.

Loading

Made from an acrylic sheet, I waterjet cut the rack and gear used for the loading mechansim forcing the ball upwards between the launch wheels. I also developed the firmware for the timing of the reload servo synchronyzed with the halo being in the correct position.

User Interface

The user interface was entirely developed by me. From creating the custom volleyball net graphic, to creating an encoded string to send to the microcontrollers with instructions for operation I developed full stack. A lot of effort was put into making the user interface as intuitive as possible. As it was used via touch screen, pressing one of the presets changed the display to show that set, with a trajectory, displayed and subsequent taps added that set to the sequence. All of this coding was self-taught throughout this project as I had never used either XAML or C# prior. Version control was also maintained via GitHub, which was essential as the computer used to run the program was not the same computer I developed the code on.

Challenges and Solutions

Custom Sets & Live Trajectory Display

I really wanted to display the trajectory of the ball. This could be done easily for the standard sets by drawing out all the options and changing the display when they get selected, but this would not work for the custom sets. I investigated many ways of doing this, and settled on creating a bezier curve. If you are unfamiliar with bezier curves, they are curves with a start and end point, as well as two control points. These control points can be used to make a parabolic shape with variable slope, perfect for displaying the trajectory of a ball.

By changing the control point's x and y positions according to the sliders, the drawing could be updated instantly. Additionally, by calculating the different between the start point and the first contol point, the slope of tangent line to launch could be calculated from the display itself. This allows the custom sets on the interface to actually translate to a calculated launch angle, and similarly the distance between control point and start point were used to calculate the required motor speed, found from baseline testing at set power levels.

Automatic Blinds | Technical Project

Project Background

As part of integrated engineering, every year we form groups around an idea and complete a term long project. We have a lot of flexibility with what we attempt to do, as long as we can show a solid game plan and market value for our project. For IGEN 230, out budget is $200.

In my 2nd year, five of my peers and I decided we wanted to make a device that helps us be well rested and adds convenience to our lives. We decided to build a device that could bring manual window blinds into the 21st centursy as an IoT enabled device. Not only reducing the effort of letting more natural light and air into a room, but also allowing scheduling so in the mornings natural light could enter at a chosen time.

Project Description

Our final product was a simple but compact attachment to the tilting mechnaism of blinds, controlled by an ESP32 powered by a portable battery bank. This system could be accessed via WiFi access point and the user could interact pick the state of the blinds from any smart device.

Motor and Mounting Bracket

To physically interact with the blinds, we had to rotate a small hook near the ceiling. This is traditionally attached to a hanging stick that when rotated controls the angle of the blind slats. This was achieved using a simple 12V motor enclosed ina wall-mounted 3D printed bracket with adjustable positions to fit mutliple kinds of blinds. A small joint hook was also 3D printed and attached to the motor shaft to act as the interface.

ESP32 and Accelerometer

Powering this system was an ESP32 microcontroller. We chose this becuase of the practical functionality of the device, as well as the low profile and power consumption. This devices features wifi and bluetooth connectivity, whichw ere both viable optiosn we wanted to explore at the inception of the project. Ultimately we settled on WiFi due to user convenience, however we were forced to use the device as an access point. Our final demonstration of the project at Design and Innovation Day 2022 took place in the Life Science building on UBC Vancouver's campus, and due to strict SSID login protocols we were not able to connect to public WiFi. Most of the functionality was developed in station mode on a private network for ease of use, but the final version was hosted directly from the ESP32-hosted webpage.

Connected to the ESP32 was a L298N H-Bridge motor controller as well as a ADXL335 accelerometer. Both of these components were chosen based on their valid functionality and ease of acquisition. The L298N controller interfaced between the ESP32 and the 12V motor. Becuase the load was so small, we were able to run the motor off of only 5 volts without any issues. The accelerometer was attached to one of the blind slats and gave positional feedbakc to the ESP32 to ensure the sleected orientation was accurate.

User Interface

As stated previously, the webpage used to control this device was hosted directly from the ESP32. This involved writing HTML and CSS code directly to the firmware of the ESP32, which was quite limmiting in terms of design and functionaily, and was definitly identified as an area of improvement for the future. A simple button selection allowed the user to directly contorl the device, as well as a secondary window where users can pick a time each day for the blinds to open and a time they close. This is meant to be scheduled with the user's normal sleep routine to allow natural light in the morning to wake them more peacefully than a sound-based alarm clock.

The user can select up to 8 sets then activate the play button, after which the machine will run through the specified sequence of sets. The dispaly also features a custom made volleyball net graphic with a live visual of the ball and it's trajectory, including for the custom sets.

Personal Contributions

This project originated from my own elevator pitch after I saw a set of automated blinds and was very jealous. As such, I had a very braod hand throughout the projects.

Mounting Bracket and Hook

I designed the brakcet used to mount the motor to the wall and the interfacing hook between motor and blinds. several gorup members contributed models for this task, but after group review mine were selected to print due to aesthetic appeal and functionality. The bracket is a simple rectiangular shape allowing the motor to slide inside prior to attaching to the wall, with many different posiitons possible through holes long the length that lien up with holes on the motor. This way the same bracket could be used for different sizes of blinds.

ESP32 and Accelerometer

I was the most hands-on with the ESP32 and other hardware for this project. Guided significantly from Random Nerd Tutorials, I created an access point system that could take input from a connected mobile user and translate it into microcontroller logic. I also implemented the accelerometer and wrote the code translating gyroscopic data into blind position.

User Interface

Since the HTML webpage was written to the ESP32 firmware, I spearheaded both endeavours. I have some basic HTML and CSS experience, but this project had some unexpected complexity getting it to function which took many unit tests and iterations. Ultimately, a functional webpage was available for viwers to connect to and control the blinds on the day of out presentation.

Frame

We needed a way of demonstarting the functionality of our project, so I contructed a simple frame out of spare wood and installed the system. As the wood was spare from our workshop, it was not the most pleasing but it was storng and fucntional, and the I base design prevented the blinds from fallign over when stood up on their own.

Challenges and Solutions

Full open and close automation of blinds

When the project first started, the plan was to fully control the opening and closign of the blinds and not just adjust the tilt. At it's inception, this functionality was not part of my own scope of work. As the project progressed, there was veryb little progress on this front and eventually we determined we needed tochange the scope of the project to focus on just the tilt, which allows for 90% of the light to enter regardless and by only controlling the tilt, the manual mechanism for opening and closing the blinds was still perfectly intact and inaccessible.

While I would have loved to have the full fucntionality, we recognised that our minimum viable product did not contain that feature so we invested more time in the tilting fucntionality. It was disppointing, but an important lesson not to get stuck underwater and to focus on delivering a solution on schedule and budget.
